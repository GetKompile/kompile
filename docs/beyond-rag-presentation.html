Part I

What is RAG?
Retrieval-Augmented Generation

RAG in 60 Seconds

Give an LLM context it wasn't trained on

The Problem: LLMs only know their training data. They can't answer questions about YOUR documents, YOUR code, YOUR company. The RAG Solution: 1. Index Take your documents, split them into chunks, convert each chunk into a vector (embedding), store in a vector database. 2. Retrieve When a user asks a question, convert the question into a vector, find the most similar chunks via cosine similarity (top-K). 3. Generate Paste those chunks into the LLM's prompt as context. The LLM answers using the retrieved text. Documents → Chunks → Embeddings → Vector Index ↑ Question → Embed → Search (top-K) → LLM + Context → Answer
For anyone unfamiliar: RAG stands for Retrieval-Augmented Generation. It's the standard pattern for giving LLMs access to private data. You take your documents, split them into chunks, embed each chunk as a vector, and store them in a vector database. When the user asks a question, you embed the question, find the most similar chunks via cosine similarity, paste those chunks into the LLM's prompt, and the LLM generates an answer based on the retrieved context. It works well for documentation Q&A, knowledge bases, and reference lookups. Every major AI framework supports it. But it has fundamental limits that show up quickly in real applications.

What RAG Does Well

Unstructured knowledge retrieval

  • Find relevant passages in large corpora
  • Answer questions from documentation
  • Surface related content by meaning
  • Handle natural language queries
Documents --> Chunks --> Embeddings | Vector Index | Query --> Embed --> Search --> Top-K | LLM + Context --> Answer

"What does our refund policy say?" → perfect RAG use case
"What's the current inventory count?" → not so much

RAG works great for documentation QA, knowledge bases, and reference lookups. The key insight is that it's a retrieval strategy -- it finds relevant text by semantic similarity. "What does our refund policy say?" is a great RAG question because the answer lives in a static document. But many applications need more than text retrieval -- they need live state, computation, and write-back.

The Core Problem

RAG retrieves information about things.
Applications need to interact with things.

RAG is read-only and probabilistic.

Real applications need read-write and deterministic access.

This is the thesis of the entire talk. RAG gives you information about things -- it retrieves relevant passages. But real applications need to interact with things -- query live state, compute values, modify records, and verify results. The shift is from retrieval to agency. To make this concrete, we need a domain where the difference between "approximately right" and "exactly right" actually matters. Where getting the wrong number isn't just annoying -- it breaks the experience.

A Virtual Tabletop is software that replaces the physical table for tabletop RPGs -- it handles maps, dice, character sheets, and rules. Elthoria is a VTT for solo and GM-less play. In traditional RPGs, a Game Master runs the world, controls NPCs, and adjudicates rules. In GM-less play, the player drives the story themselves, using oracle systems like the Mythic Game Master Emulator to answer narrative questions: "Does the guard notice me?" becomes a fate check against the scene's chaos factor. It helps update game state -- rolling dice with real modifiers, applying damage, tracking buffs and spell slots, managing initiative order -- so the player can focus on the story. When the oracle says "an unexpected NPC arrives", the AI can narrate that in character, pull from the monster database, generate a stat block, and place the NPC on the battle map. But the oracle drove the decision, not the AI. This separation matters architecturally: the AI needs structured read/write access to game state (196 MCP tools), but the narrative direction comes from randomized oracle tables -- deterministic game mechanics meeting emergent storytelling. Four CLI agents are available with automatic rate-limit fallback so play isn't interrupted if one provider hits its API limit.

  • Spell slots → resource quotas, rate limits
  • Initiative order → workflow state machines

If RAG can't handle a tabletop RPG, it can't handle your enterprise data either.

Tabletop RPGs are the perfect stress test because they demand everything RAG can't do -- simultaneously. You need exact numbers (HP is 42, not "around 40"), complex rule interactions (does this bonus stack with that one? Pathfinder 1e has 13 bonus types with specific stacking rules), live mutable state (HP changes every round, buffs expire on turn count, characters move on a grid), spatial reasoning (is the target within 30 feet? is there line of sight?), and write-back (the AI GM needs to actually deal damage, not just describe it). But here's the key: every one of these maps directly to enterprise problems. HP tracking is inventory management. Bonus stacking is tax code or pricing engine logic. Battle maps are floor plans or network topologies. Spell slots are resource quotas. Initiative order is a workflow state machine. If your AI architecture can handle Pathfinder 1e, it can handle your ERP system.

Where RAG Falls Apart

When you need exact answers, not similar ones

Scenario RAG gives you You actually need
"What is Aldric's current HP?" A passage mentioning HP 42 (from live DB)
"Roll a d20 + attack bonus" Text about attack rules 17 (actual dice roll + modifiers)
"Show the current battle map" A description of a map Live ASCII grid with positions
"Apply Bless to the party" The Bless spell description State mutation + bonus tracking
Now that you know the domain, watch RAG fail at every row. Aldric's HP changes every round of combat -- vector embeddings are stale the moment damage is dealt. Dice rolls require actual randomness with character-specific bonuses computed from the Pathfinder rules engine -- you can't retrieve a dice roll from a document. The battle map is a live data structure with entity positions on a coordinate grid that changes mid-combat. Applying the Bless spell is a state mutation: it adds a +1 morale bonus to attack rolls and saves for the entire party, tracked via the UnifiedBonus engine with type-specific stacking rules. RAG would give you the Bless spell description from the rulebook. The game needs the bonus actually applied to every affected character's computed stats.

Five Limits of Pure RAG

  1. 1 No live state — embeddings are a snapshot, not current truth (get_character_summary)
  2. 2 No relationships — flat vector space loses graph structure (graph_find_connected)
  3. 3 No write-back — can't modify the world, only read about it (update_hit_points)
  4. 4 No computation — can't roll dice, calculate bonuses, run formulas (roll_attack, UnifiedBonus)
  5. 5 No guarantees — "most similar" != "correct answer" (HallucinationEvaluator)
These five limits aren't theoretical -- they show up immediately when you try to build anything beyond a documentation chatbot. The Elthoria case study makes these visceral: you can't approximate HP, you can't hallucinate dice rolls, you can't describe a map that doesn't exist, and you definitely can't let the AI modify game state by generating text about it. Each parenthetical on the right is an actual Elthoria MCP tool that solves that specific limit -- we'll see these in the next section.

The UnifiedBonus Engine

Why an LLM can't compute Pathfinder bonuses from text alone

// UnifiedBonus.java - every modifier in the game
public class UnifiedBonus {
    String name;
    UnifiedBonusType type;     // ARMOR, DODGE, SHIELD,
                               // ENHANCEMENT, DEFLECTION,
                               // NATURAL_ARMOR, MORALE...
    BonusSourceType source;    // FEAT, ITEM, BUFF,
                               // CLASS_FEATURE,
                               // ABILITY_MODIFIER
    int value;
    boolean stackable;
}

// Pathfinder stacking rules:
// - DODGE bonuses always stack
// - Same-type bonuses: highest wins
// - Different-type bonuses: all apply
// - Circumstance bonuses stack
// - Untyped bonuses stack
Aldric's AC Breakdown: Base: 10 + Armor (chain shirt): +4 ARMOR + Shield (heavy steel): +2 SHIELD + DEX modifier: +2 ABILITY + Ring of Protection: +1 DEFLECTION + Amulet of Nat Armor: +1 NATURAL + Shield of Faith (buff): +2 DEFLECTION ^ Ring SUPPRESSED: Shield of Faith +2 > Ring +1 Same-type bonuses: only highest applies! ───────────────────────────── Total AC: 21 (not 22) An LLM adding these up would get 22. The rules engine knows to take max(+1, +2) for DEFLECTION, giving 21.
This is the single strongest argument for structured computation over RAG. Pathfinder's bonus stacking rules are arcane -- same-type bonuses don't stack (highest wins), except dodge and circumstance which always stack. An LLM reading the rules text will get this wrong regularly. The UnifiedBonus engine applies these rules deterministically. Every feat, item, buff, and class feature registers bonuses with their type, and the engine resolves conflicts automatically. The MCP tool calculate_armor_class returns the breakdown showing exactly which bonuses applied and which were suppressed.

The Grid DSL (ANTLR4)

A formal grammar for map definitions — parseable, not prose

grammar TTRPGMap;

dungeon : section+ ;

section
    : header | gridDef | grid | party | entityStats
    | entities | encounters | traps | puzzles | loot
    | obstacles | rooms | passages | terrain | biome ;

// Entity with stat reference, placed at coordinates
entityPlacementDetailed
    : entityId COLON position entityRef entityInstanceProps? ;

entityRef
    : STAT_LITERAL entityStatId     // -> ENTITY_STATS block
    | CR_LITERAL crValue            // Quick CR-only placement
    | NPC_LITERAL NUMBER ;          // -> DB NPC by ID

// Fully structured stat block definition
entityStatDef
    : entityName cr=challengeRating hp=hitPoints
      ac=armorClass size=creatureSize
      type=creatureType alignment=creatureAlignment ;
The ANTLR grammar means maps are formally parseable, not just text. The entityRef rule shows three ways to place a creature: reference a stat block defined in the same map file, specify a CR for quick placement, or reference an NPC by database ID. This bridges the gap between the map DSL and the structured database -- you can place NPC ID 42 at position 5,3 on the grid, and the AI can query that NPC's full stats via get_npc.

Grid Rendering: Deterministic Overlay

Same state → same output. Every symbol has strict precedence.

// Grid.java toString() - the rendering engine
// For each cell at (x, y):
char displayChar = cells[y][x]; // Base terrain

// Overlay in strict precedence order:
if (entities.containsKey(pos))
    displayChar = entities.get(pos).getSymbol();
else if (traps.containsKey(pos))     displayChar = 'T';
else if (puzzles.containsKey(pos))   displayChar = 'P';
else if (loot.containsKey(pos))      displayChar = 'L';
else if (objects.containsKey(pos))   displayChar = 'O';
else if (obstacles...)               displayChar = 'F';
else if (passages...)                displayChar = p.getSymbol();
Precedence (highest to lowest): 1. Entities (creatures, PCs) 2. Traps (T) 3. Puzzles (P) 4. Loot (L) 5. Objects (O) 6. Obstacles (F) 7. Passages (doors, stairs) 8. Terrain (floor, wall, water) Entities always on top. Same grid state = same ASCII output.
The rendering is deterministic -- same grid state always produces the same ASCII output. The precedence chain is the key: entities always overlay terrain, traps show above loot, etc. This isn't just a visualization trick -- it means the AI can trust that the ASCII representation is always consistent with the underlying data. Call get_grid_ascii, read the character at position (5,3), and know exactly what's there.

Part V

The 4-Layer Grounding Architecture

How Context Reaches the AI

Four complementary layers, each solving a different problem

Layer What How When
1 Mode-Aware Context Injection 5 ContextConfigs → DB queries → markdown Every request
2 Keyword Entity Matching Substring scan → entity snippets (2K cap) When entities mentioned
3 Budget-Capped World Assembly Priority-ordered parallel fetch, 12K char cap Plot & roleplay modes
4 MCP Tool Self-Retrieval 196 tools → on-demand DB queries by agent During reasoning

Plus: Kompile RAG for rulebook lookups via rag_query_rules — reference material, not game state

Four layers, each solving a different problem. Layer 1 runs on every request -- five mode-aware configs (default, combat, roleplay, conversation, plot designer) select different boolean flags that control which database queries fire. Layer 2 is reactive -- EntityKeywordMatcherService builds an in-memory index from all entity names and does case-insensitive longest-keyword-first substring scanning, capped at 2,000 characters. Layer 3 uses PlotContextBuilderService to parallel-fetch world data via forkJoin with a hard 12K character budget and priority-ordered assembly (NPCs first, then factions, threads, locations, events). Layer 4 is the agent itself -- 196 MCP tools give it direct database access for anything not pre-injected. Graph Construction Pipeline

Documents | v +------------------+ +-------------------+ | Chunk Documents |---->| LLM Extraction | Per-chunk entity/relation extraction +------------------+ | (structured JSON) | via GraphExtractionSchema +--------+----------+ | +--------v----------+ | Entity Resolution | Merge duplicates across chunks | Service | (same entity, different mentions) +--------+----------+ | +-------------+-------------+ | | +--------v--------+ +--------v--------+ | Neo4j Graph | | JPA Knowledge | | (Cypher + | | Graph (SQL + | | APOC MERGE) | | graph algos) | +-----------------+ +-----------------+
// Neo4jGraphConstructor.java - Cypher upsert
private static final String UPSERT_NODE_QUERY = """
    UNWIND $rows AS row
    MERGE (n {id: row.id})
    ON CREATE SET n = row.properties, n.id = row.id
    ON MATCH SET n += row.properties
    WITH n, row.labels AS labels
    CALL apoc.create.addLabels(n, labels) YIELD node
    RETURN elementId(node) AS elementId""";
The pipeline: documents are chunked, each chunk goes to an LLM for entity/relation extraction using the GraphExtractionSchema format, then EntityResolutionService merges duplicate entities across chunks. The results can go to either Neo4j via APOC Cypher MERGE or to the JPA-backed knowledge graph. The MERGE query is idempotent -- running it twice with the same entity updates properties instead of creating duplicates.

ReAct Agent: Graph-Grounded Reasoning

// kompile-react-agent - Reason-Act-Observe loop

// Three specialized reasoners:
DefaultReasoner        // Standard LLM reasoning
GraphRagReasoner       // Reasoning grounded in graph retrieval
EvalBasedReasoner      // Evaluation-in-the-loop reasoning

// The loop:
while (!done) {
    // REASON: analyze state, decide next action
    Action action = reasoner.reason(context, memory);

    // ACT: execute tool call (sequential or parallel)
    Result result = actor.execute(action);

    // OBSERVE: parse result, update memory
    Observation obs = observer.observe(result);
    memory.add(obs);

    // Check: is the answer sufficient?
    done = finalizer.shouldFinalize(memory);
}

// GraphRagReasoner augments each reasoning step with:
//   - Entity context from knowledge graph
//   - Relationship paths between mentioned entities
//   - Community summaries from Louvain detection
The ReAct agent framework closes the loop. GraphRagReasoner doesn't just retrieve documents -- it pulls entity context and relationship paths from the knowledge graph at each reasoning step. EvalBasedReasoner runs evaluation after each step to check for hallucination before continuing. These aren't toy patterns -- they're production reasoning loops that combine structured data access with LLM reasoning.

RAG Tool: Where It Fits

// RagToolImpl.java - RAG as one tool among hundreds

@Tool(name = "rag_query",
    description = "Queries the document corpus using the configured
                   retriever and returns relevant information snippets.")
public Map<String, Object> executeRagQuery(RagQueryInput input) {
    // Optimization: cap docs and content length
    int maxDocs = cfg.getRagMaxDocs();  // default: 3

    List<RetrievedDoc> docs =
        documentRetriever.retrieveWithDetails(input.query(), maxDocs);

    // Handle multi-vector docs (tables get full content, not summary)
    for (RetrievedDoc doc : docs) {
        String content = extractFullContent(doc);

        // Large results become fetch handles, not inline blobs
        if (content.length() > maxContentChars) {
            String resultId = resultCache.store(fullPayload);
            result.put("result_id", resultId);
        }
    }
    return result;
}

RAG is just one tool. The AI chooses when it needs reference material vs. live data.

Notice the optimizations here -- response capping, result-reference caching, multi-vector table handling. Even RAG itself benefits from structured thinking. Tables get their full content returned instead of just the embedding summary. Large results become fetch handles so the AI can page through them. RAG is powerful, but it's one tool in a 642-tool arsenal.

MCP Optimization: Scaling 642 Tools

// McpOptimizationConfig.java - three meta-tool modes

enum MetaToolMode {
    DIRECT,   // Expose all 642 tools (huge context cost)
    DYNAMIC,  // Only expose: search_tools, describe_tools,
              //   execute_tool + allow-list
    HYBRID    // Meta-tools + rag_query + read_file + list_files
}

// Three optimization layers:
// 1. Surgical caps on large responses
ragMaxContentChars = 2000;    // per document
ragMaxDocs = 3;               // per query
knowledgeGraphTruncateChars = 200;

// 2. Result-reference cache (fetch handles for large payloads)
compressionThresholdChars = 4000;
// -> Returns: { "result_id": "abc123" }
// -> AI calls: fetch_result("abc123") to page through

// 3. Per-tool overrides
toolOverrides = { "rag_query": { maxResponseChars: 1500 } };
Scaling 642 tools is a real problem -- you can't put 642 tool schemas in context. The meta-tool pattern solves this: expose only search_tools, describe_tools, and execute_tool. The AI discovers what it needs on demand. HYBRID mode adds a small whitelist of frequently-used tools like rag_query and read_file that bypass discovery. The three optimization layers -- response caps, result caching, and dynamic toolsets -- work together to keep context manageable.

DYNAMIC Mode in Action

AI sees 3 meta-tools instead of 642 — discovers on demand

User: "What entities are connected to Acme Corp?" AI's visible toolset (DYNAMIC mode): search_tools, describe_tools, execute_tool Step 1: search_tools({ query: "graph entity connected" }) -> ["graph_find_connected", "graph_search_by_entity", "graph_get_path", "graph_get_overview", "graph_get_related_documents"] Step 2: describe_tools({ tools: ["graph_find_connected"] }) -> { name: "graph_find_connected", params: { node_id: "string", max_depth: "int (default 2)" }, returns: "connected nodes with relationship types" } Step 3: execute_tool({ tool: "graph_find_connected", args: { node_id: "acme-corp", max_depth: 2 } }) -> { nodes: [{ id: "john-doe", type: "PERSON", relation: "CEO_OF" }, { id: "widget-inc", type: "ORG", relation: "PARTNER_OF" }] } Result: AI found and used the right tool without ever seeing its schema in context.
This is what DYNAMIC mode looks like in practice. The AI starts with only 3 tools in context -- search_tools to find tools by keyword, describe_tools to get the JSON schema for a specific tool, and execute_tool to call it with arguments. Three tool schemas instead of 642. The AI discovers what it needs on demand. The context savings are massive -- 642 tool schemas might consume 50K+ tokens; 3 meta-tool schemas use under 500 tokens. The trade-off is latency -- 3 tool calls instead of 1 -- but the context savings make longer conversations possible. Note: Kompile's meta-tools (search_tools/describe_tools/execute_tool) are distinct from Elthoria's discovery tools (list_all_tools/search_tools/get_tool_details) shown earlier -- same pattern, independent implementations. Elthoria's discovery runs within a flat tool namespace; Kompile's meta-tools wrap the entire tool execution layer.

Guardrails & Evaluation
Trust but verify

Tool Gateway: LLM-Based Policy

// ToolGatewayService.java - policy enforcement on EVERY tool call

public GatewayDecision evaluate(String toolName, Map<String, Object> args) {
    List<ToolGatewayRule> matchingRules = rulesProvider.getMatchingRules(toolName);
    if (matchingRules.isEmpty()) return GatewayDecision.allow();

    // LLM evaluates the tool call against matched rules
    GatewayDecision decision = evaluateWithLlm(toolName, args, matchingRules);
    // Returns: ALLOW, REWRITE (sanitize args), or BLOCK

    if (properties.isDryRun()) {
        log.info("[DRY-RUN] Would have {} '{}'", decision.action(), toolName);
        return GatewayDecision.allow();
    }

    return decision;
}

// Example rules:
// - "Block any write_file to paths containing 'config' or '.env'"
// - "Rewrite PII in query arguments before passing to RAG"
// - "Block all tool calls from unauthenticated sessions"
The Tool Gateway is an LLM-in-the-loop policy enforcer. Before any tool executes, matching rules are sent to a separate LLM that decides: allow, rewrite the arguments, or block. This enables data governance -- block writes to production, sanitize PII from query arguments, enforce access control. It supports dry-run mode for testing policies and fail-open or fail-closed on errors.

Hallucination Detection

// HallucinationEvaluator.java

private static final String EVALUATION_PROMPT = """
    Analyze the response for hallucinations - claims not
    supported by the context.

    For each factual claim, check if it can be verified.
    A hallucination is a claim that:
    1. Cannot be verified from context documents
    2. Contradicts information in the context
    3. Adds specific details not present in the context

    Calculate: score = 1.0 - (hallucinated / total_claims)
    """;

// Returns:
{ "score": 0.85,
  "totalClaims": 7,
  "hallucinatedClaims": 1,
  "hallucinations": ["Claimed the company was founded in 2008,
                      but context says 2010"],
  "explanation": "6 of 7 claims verified..." }
Hallucination detection is an LLM-as-judge evaluator. It takes the response and the context documents, identifies every factual claim, and checks each one against the context. Claims that can't be verified, contradict the context, or add unsupported details are flagged. The score is 1 minus the ratio of hallucinated claims to total claims. This runs continuously on AI responses, not just at test time.

The Hybrid Pattern
Complementary, not competing

When to Use What

Use Case Approach Why
"What does the Bless spell do?" RAG Reference text, static content
"What is Aldric's current HP?" Tool Live state, exact value needed
"Who is connected to Acme Corp?" Graph Relationship traversal
"Roll to attack the goblin" Tool Computation + state mutation

This table is the practical decision framework. RAG for reference text and static content. Tools for live state, computation, and mutations. Graphs for relationship traversal and structural queries. Structured injection for ordered, prioritized data. The answer is almost never "just use RAG" -- it's "use the right tool for the data shape." And in most real applications, you need all of these working together.

This is the complete picture. User query flows through context assembly (4 grounding layers), then to LLM reasoning with ReAct tool discovery. The LLM can reach into five different backends: vector index for RAG, Neo4j/JPA for graph queries, code index for structural code navigation, game state for live data, and filesystem for sandboxed file access. Every tool call passes through the Tool Gateway for policy enforcement. Evaluation and guardrails run continuously on outputs. This is the hybrid pattern -- RAG, tools, graphs, and guardrails working together.

Key Takeaways

  1. 1 RAG is a retrieval strategy, not an architecture
    It's one tool in a toolkit — not the toolkit itself
  2. 2 Structured data gives deterministic guarantees
    When you need exact answers, query a database, not a vector index
  3. 3 MCP tools give AI agency over live state
    Read, write, compute, and rollback — not just retrieve
  4. 4 Graphs capture what vectors can't: relationships
    Entity connections, call chains, causal paths — structure matters
  5. 5 The hybrid pattern is the answer
    RAG for reference, tools for state, graphs for relationships, guardrails for trust
Five takeaways to leave them with. First: RAG is a strategy, not an architecture -- stop treating it as the whole solution. Second: structured data wins when you need exactness -- the UnifiedBonus example shows this viscerally. Third: MCP tools transform AI from a reader to an actor -- read, write, compute, rollback. Fourth: graphs are the missing middle between unstructured text and relational databases -- they capture what vectors flatten. Fifth: the hybrid pattern is the real answer -- use all of these together, each for what it does best. Elthoria proves this works at scale: 196 tools, 4 grounding layers, rule enforcement, multi-agent fallback, and RAG as a reference tool -- not the foundation.
