Knowledge and Action Automation (KAA)

An operationalizable methodology for automating the modeling of work, not just the work itself. Six layers, three governance planes, one closed feedback loop, and a cross-customer flywheel.

KAA splits the problem along a deliberate seam. KNOWLEDGE layers (L1–L3) produce claims about how work happens and how it should happen. ACTION layers (L4–L5) produce executable connections and agents that do the work. The seam between them is where governance, operating model, and observability concerns must be made explicit — otherwise the methodology cannot survive contact with a real customer.

The closed-loop feedback the original 5-layer stack didn't show.
Layer 5 telemetry — corrected decisions, recurring exception patterns, confidence-vs-outcome divergence — flows back into Layer 2 (ontology refinement) and Layer 1 (re-evidence). Without this loop, KAA is a one-shot consulting deliverable. With it, KAA is a learning system. The single highest-leverage operational addition.
Section 2

Process-graph view — Layer 3, redrawn for the real world

Real finance is a dependency graph, not a single workflow.

Every Layer-3 process model in production has upstream and downstream dependencies. The single-process abstraction breaks on customer two. KAA models the graph, not just the node.

The demo's "monthly close + 3-mo P&L" (purple) is one node. Real engagements must model the upstream dependencies (procurement, sales, GR, payroll), the parallel close streams (3-way match, AP close, revenue close, inventory close, FX), and the downstream consumers (variance commentary, board pack, tax, external reporting). Layer 3 must support this graph, including shared ontology across nodes and cross-process escalation paths.

Section 3

The cross-customer flywheel — how KAA becomes a platform

Asset accumulation across customers

The platform thesis: customer-N is faster than customer-(N−1) because the shared knowledge base compounds. Without an explicit accumulation policy, every customer is a new project.

What contributes, what compounds

Ontology templates (Layer 2): domain-shaped (FP&A, P2P, OTC, ITGC, R&D capitalization, lease accounting). Customer-N starts from a 70%-accurate template instead of zero.

Anonymized eval sets (Layer 5): "what good agent behavior looks like" for variance triage, exception routing, commentary drafting. Customer-N's agents are quality-graded against accumulated industry baselines.

Connector implementations (Layer 4): NetSuite, SAP, Workday, Salesforce, Shopify, Coupa. Once built, the marginal cost is near zero.

Process patterns (Layer 3): "the close-cycle pattern" or "the P2P three-way match pattern" — re-applicable shapes, customer-specific instances.

What does NOT compound (and must not): customer data, customer-specific evidence, anything PII or trade-secret. The IP firewall is part of the moat — without it, no customer signs.

Operationalization gaps — addressed, named, and still ahead

✓ Addressed in this view

  • Layer 0 for engagement, scoping, baseline metrics
  • Three planes: governance, operating model, observability
  • Closed-loop feedback from Layer 5 to Layers 2 and 1
  • Knowledge / Action seam made visually explicit

◆ Named, not yet implemented

  • Discovery completion criterion — coverage of named roles, evidence saturation, conflict-resolution rate
  • HITL fatigue management — queue prioritization, batch approval, threshold tuning, circuit breakers
  • Replay / counterfactual sandbox — Layer 3 needs a "what would have happened" mode
  • Drift detectors per layer — when does the model need re-discovery?
  • Methodological specificity per layer — what discipline lives in each (BPMN/CMMN/DMN, OWL/DDD, agent-spec format)
  • Per-layer unit economics — cost driver, pricing posture

→ Future work

  • Integration with PMM estate — Signavio, Workiva, Confluence, Collibra: read & write into the customer's existing process-management investment
  • Liability / SLA framework — explicit boundaries and incident-response playbooks per agent class
  • Continuous-discovery automation — triggered by drift, customer events (M&A, ERP migration, reorg)
  • Multi-tenant ontology federation — how customer-specific specializations co-exist with shared templates without leaking IP
  • Industry / domain plug-in mechanism — how new verticals (healthcare, legal, public sector) get added without forking the core
  • Regulatory-framework mappings — explicit traceability from each layer's outputs to SOX, J-SOX, EU AI Act, NIST AI RMF, ISO 27001 evidence requirements
