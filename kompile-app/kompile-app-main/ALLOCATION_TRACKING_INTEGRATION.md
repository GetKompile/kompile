# Allocation Tracking Integration (ADR 53)

## Overview

This document describes the integration of ADR 53 (NDArray and OpContext Allocation Logging) into the Kompile application, providing a comprehensive diagnostic endpoint that consolidates ALL debugging information in one place.

**Key Point**: ADR 53 allocation logging is **automatically active** in functrace builds (`-Dlibnd4j.calltrace=ON`). There are no runtime toggles - it's always on when built with `SD_GCC_FUNCTRACE`. This integration provides read-only endpoints to view and analyze the allocation logs.

## What Was Implemented

### 1. Allocation Log Parser Service

**Location**: `ai.kompile.app.diagnostics.service.AllocationLogParser`

Parses ADR 53 allocation log files generated by the `AllocationLogger` in libnd4j when `SD_GCC_FUNCTRACE` is enabled.

**Features**:
- Parses both NDArray and OpContext allocations
- Extracts all metadata: size, shape, operation, timestamps, stack traces
- Calculates allocation statistics and hotspots
- Identifies top allocation sites and operations
- Computes allocation rates (allocations/sec, MB/sec)

### 2. Data Transfer Objects (DTOs)

**Location**: `ai.kompile.app.diagnostics.dto`

**AllocationEntry**:
- Represents a single allocation (NDArray or OpContext)
- Contains all metadata from the log entry
- Includes C++ stack trace information

**AllocationLogSummary**:
- Aggregated summary of all allocations
- Top allocations by size
- Allocations grouped by operation
- Top allocation sites by frequency
- Rate statistics

**AllocationRateStats**:
- Allocation rate calculations
- Time-based metrics
- Memory growth rate

### 3. Comprehensive Diagnostics Controller

**Location**: `ai.kompile.app.web.controllers.ComprehensiveDiagnosticsController`

**Main Endpoint**: `/api/diagnostics/comprehensive`

Returns ALL diagnostic information in a single call:
- System information (JVM, OS, memory usage)
- Lifecycle tracking configuration (ADR 51)
- Cache statistics (Shape and TAD caches)
- Allocation logging summary (ADR 53)
- Operation execution logging status (ADR 52)
- Leak reports location and status

## Usage Examples

### Example 1: Quick Health Check

```bash
# Get comprehensive diagnostics with summary only
curl http://localhost:8080/api/diagnostics/comprehensive | jq '.summary'

Output:
{
  "lifecycleTracking": "true",
  "allocationLogging": "true",
  "totalAllocations": "5000",
  "totalAllocationsMB": "100.00",
  "cacheMemoryUsageMB": "4.30"
}
```

### Example 2: Identify Memory Hotspots

```bash
# Get top 50 allocations and sites
curl 'http://localhost:8080/api/diagnostics/comprehensive?topAllocations=50&topAllocationSites=20' | \
  jq '.allocationLogging.summary.topAllocationSites'

Output:
{
  "sd::ops::matmul::execute at matmul.cpp:67": 1200,
  "sd::ops::conv2d::execute at conv2d.cpp:45": 800,
  "sd::NDArray::NDArray at NDArray.hXX:350": 500
}
```

### Example 3: Analyze Allocation Rates

```bash
# Check memory growth rate
curl http://localhost:8080/api/diagnostics/allocation-log/summary | \
  jq '.rateStats'

Output:
{
  "firstAllocationTime": 1737569535000000000,
  "lastAllocationTime": 1737569540000000000,
  "durationMs": 5000,
  "allocationsPerSecond": 1000.0,
  "bytesPerSecond": 20971520.0,
  "mbPerSecond": 20.0
}
```

### Example 4: Find Largest Allocations

```bash
# Get the 10 largest allocations
curl http://localhost:8080/api/diagnostics/allocation-log/summary | \
  jq '.largestAllocations[:10] | .[] | {size: .sizeBytes, operation: .operation, shape: .shape}'

Output:

  "size": 4000000,
  "operation": "matmul",
  "shape": [1000, 1000]
}
{
  "size": 3600000,
  "operation": "conv2d",
  "shape": [100, 100, 9, 4]
}
```

### Example 5: Complete Diagnostic Dump

```bash
# Get everything including detailed entries
curl 'http://localhost:8080/api/diagnostics/comprehensive?includeAllocationDetails=true' > diagnostic_dump.json

# Pretty print and analyze
jq '.allocationLogging.detailedEntries | group_by(.operation) | map({operation: .[0].operation, count: length}) | sort_by(.count) | reverse' diagnostic_dump.json
```

## Integration with Existing Systems

This implementation integrates with:

### 1. Lifecycle Tracking (ADR 51)
- Accesses lifecycle tracking configuration
- Shows enabled features and settings
- Complements leak detection with allocation patterns

### 2. Operation Execution Logging (ADR 52)
- Checks for operation execution log files
- Provides path and status information
- Helps correlate operations with allocations

### 3. Cache Statistics
- Reports Shape and TAD cache usage
- Shows current and peak memory consumption
- Helps identify cache-related memory issues

### 4. Model Debugging
- Can be used alongside ModelDebugController
- Provides allocation context for model operations
- Helps identify which model operations allocate the most

## Building with Allocation Logging

> **Key Principle**: Allocation logging is a **build-time feature**, not a runtime toggle.
> When you build with `SD_GCC_FUNCTRACE`, allocation logging is **always active** - no need to enable it.

To enable allocation logging, build with `SD_GCC_FUNCTRACE`:

```bash
# Native build with functrace
cd libnd4j
./buildnativeoperations.sh -DLIBND4j_CALLTRACE=ON

# Or via Maven
mvn clean install -Dlibnd4j.calltrace=ON -Dlibnd4j.build=debug
```

**Result**: All NDArray and OpContext allocations are automatically logged to `/tmp/nd4j_allocations_<PID>.log`

## Environment Variables

```bash
# Optional: Set custom log directory
export SD_ALLOCATION_LOG_DIR=/var/log/nd4j

# Default: /tmp/nd4j_allocations_<PID>.log
```

## Performance Considerations

- **Allocation logging** (ADR 53): Moderate overhead, only active in functrace builds
- **Lifecycle tracking** (ADR 51): Low to high overhead depending on settings
- **API endpoints**: Fast summary, slower with detailed entries

**Recommendations**:
- Use summary endpoints for production monitoring
- Use detailed endpoints only for debugging specific issues
- Enable lifecycle tracking with "minimal" or "balanced" presets
- Allocation logging is debug-only (functrace builds)

## Troubleshooting

### Allocation Logging Not Available

**Symptom**: `"enabled": false` in allocation logging section

**Solution**: Build with `-Dlibnd4j.calltrace=ON` to enable `SD_GCC_FUNCTRACE`

### Empty Allocation Log

**Symptom**: Log file exists but has 0 allocations

**Solution**:
1. Ensure application has run some operations
2. Check if log file is being written to correct location
3. Verify `SD_GCC_FUNCTRACE` is defined at compile time

### Large Log Files

**Symptom**: Allocation log file growing very large

**Solution**:
1. Logs are session-based (one per PID)
2. Old logs are auto-deleted on startup
3. Use summary endpoints instead of raw log
4. Consider reducing `topAllocations` and `topAllocationSites` parameters

### High Memory Usage

**Symptom**: Comprehensive endpoint uses too much memory

**Solution**:
1. Don't set `includeAllocationDetails=true` in production
2. Use specific endpoints (`/allocation-log/summary`) instead
3. Limit `topAllocations` and `topAllocationSites` parameters
4. Use raw log endpoint with `maxLines` parameter

## Example Workflow: Debugging Memory Growth

> **Note**: Allocation logging (ADR 53) is **always active** in functrace builds - no enable/disable needed.
> The step below enables lifecycle tracking (ADR 51) for leak detection, which is a separate but complementary system.

1. **Enable lifecycle tracking** (optional - for leak detection):
```bash
curl -X POST 'http://localhost:8080/api/lifecycle/enable'
```

2. **Run your application** with problematic workload
   - Allocation logging happens automatically (if built with `-Dlibnd4j.calltrace=ON`)

3. **Get comprehensive diagnostics**:
```bash
curl 'http://localhost:8080/api/diagnostics/comprehensive?topAllocations=50' > diagnostics.json
```

4. **Analyze hotspots**:
```bash
# Top allocation sites
jq '.allocationLogging.summary.topAllocationSites' diagnostics.json

# Largest allocations
jq '.allocationLogging.summary.largestAllocations[] | {size: .sizeBytes, op: .operation, shape: .shape}' diagnostics.json

# Operations causing most allocations
jq '.allocationLogging.summary.allocationsByOperation' diagnostics.json
```

5. **Check allocation rate**:
```bash
jq '.allocationLogging.summary.rateStats' diagnostics.json
```

6. **Investigate specific operations**:
```bash
# Get all allocations for a specific operation
curl 'http://localhost:8080/api/diagnostics/allocation-log/entries' | \
  jq '.[] | select(.operation == "matmul")'
```

7. **Generate leak reports**:
```bash
curl -X POST 'http://localhost:8080/api/lifecycle/trigger-cleanup'
```

This workflow helps identify:
- **What** is allocating memory (operations, sizes, shapes)
- **Where** allocations happen (source locations, stack traces)
- **When** allocations occur (timestamps, rates)
- **How much** memory is being allocated (bytes, MB/sec)

## Summary

The comprehensive diagnostics endpoint simplifies gcc functrace allocation tracking by providing:

✅ **Single endpoint** for all diagnostic information
✅ **Automatic log parsing** and analysis
✅ **Hotspot identification** (top operations, sites)
✅ **Rate calculation** (allocations/sec, MB/sec)
✅ **Integration** with existing tracking systems
✅ **Multiple output formats** (JSON, raw text)
✅ **Flexible detail levels** (summary vs. detailed)

This makes debugging memory issues much easier and more efficient!
