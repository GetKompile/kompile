## Issues & Edge Cases Found

### ⚠️ Issue 1: `StdioQuorumTaskTool` does NOT use role-based fallback

**Lines 238-244 of `McpStdioCommand.java`:**
```java
var quorumTool = new StdioQuorumTaskTool(agentRegistry, subagentRunner, om, wd);
```
**No `roleManager` is passed.** The quorum tool accepts a `role` parameter (line 76 of its schema) and passes it through to `runAgent()` for `AgentConfig.roleName`, but it does NOT perform rate-limit fallback. The caller explicitly specifies which agents to query in parallel, so if any of them are rate-limited, that agent simply fails with "Rate limited." — no fallback occurs.

**This is arguably intentional** — quorum is about getting parallel opinions from specific agents, not falling back. But it's an inconsistency: `task` and `multi_task` gracefully degrade on rate limits, while `quorum_task` does not. Worth documenting or considering whether quorum should also have a fallback mechanism.

### ⚠️ Issue 2: `BuiltInRoles` don't define `agentFallbackPriority`

None of the 6 built-in roles (`CODER`, `ARCHITECT`, `REVIEWER`, `RESEARCHER`, `DEVOPS`, `DATA_SCIENTIST`) set an `agentFallbackPriority`. This means all built-in roles fall through to the global `DEFAULT_FALLBACK_ORDER`. This is fine for now, but if different roles should prefer different fallback agents (e.g., `architect` might prefer `claude` first for design reasoning), those built-in roles would need to be updated.

### ⚠️ Issue 3: `updateRole()` does NOT preserve `agentFallbackPriority`

**Lines 187-200 of `RoleManager.java`:**
```java
RoleConfig updated = RoleConfig.builder()
    .name(name)
    .displayName(displayName != null ? displayName : existing.getDisplayName())
    ...
    .modelHint(existing.getModelHint())
    .sourceFile(existing.getSourceFile())
    .isBuiltIn(false)
    .build();
```
**Missing:** `.agentFallbackPriority(existing.getAgentFallbackPriority())`, `.canSpawnSubagents()`, `.enabledTools()`, `.permissionOverrides()`, `.maxSteps()`. 

Actually, `.enabledTools()`, `.permissionOverrides()`, and `.maxSteps()` ARE preserved (lines 193-195), and `.canSpawnSubagents()` is preserved (line 196). But **`agentFallbackPriority` and `modelHint`** — wait, `modelHint` IS preserved (line 197). **`agentFallbackPriority` is NOT preserved.** This is a bug. If a user edits a role that has a custom `agentFallbackPriority`, it will be lost after update.

### ⚠️ Issue 4: No validation of agent names in `agentFallbackPriority`

The `RoleLoader` parses any comma-separated string into the fallback list without validating that the agent names are known agents. If a user writes `agent_fallback: chatgpt, bard`, those invalid names will be in the fallback list and will silently fail at runtime (caught as "not found in PATH"). Could benefit from a warning during load time.

### ⚠️ Issue 5: `toMarkdown()` doesn't serialize `sourceFile` or `isBuiltIn`

This is by design (those are metadata, not user-editable), but it means round-tripping a role through `toMarkdown()` → `parseRoleFile()` will lose the `sourceFile` path. The `saveRole()` method computes a new path anyway, so this is acceptable.
